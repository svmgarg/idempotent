üìñ Overview

The Idempotency Service is your safety net against duplicate triggers.

In distributed systems, the same request can arrive multiple times due to network failures, user retries, automation tool retries, or client-side issues. Without idempotency, this means the same operation gets executed multiple times‚Äîresulting in duplicate charges, double-booked seats, duplicate orders, or duplicate notifications.

This service identifies when the same request is sent twice (detects duplicate triggers) so your application can:

  • ‚úÖ Detect that a request is a duplicate (same unique ID arriving again)
  • ‚úÖ Skip re-executing the operation
  • ‚úÖ Return your previously cached result

How It Works:
  1. First Request: Your app calls /check with a unique ID ‚Üí Returns isDuplicate: false
  2. Process: Your app executes the operation (payment, order, etc.) and caches the result
  3. Duplicate Request: Same ID arrives again ‚Üí Calls /check ‚Üí Returns isDuplicate: true
  4. Skip: Your app returns the cached result WITHOUT re-executing
‚è±Ô∏è Memory Window: We track operation IDs for a configurable time period (default: 1 hour). After this window expires, we forget the ID and treat new requests with that ID as new operations.

‚úÖ Real-World Scenarios

Here are common situations where duplicate triggers occur and how this service helps:

1) üõí Order Creation (Most Common)

Scenario: A user clicks "Place Order" twice or the client retries due to a timeout.

Without idempotency: two orders can be created, customer may be charged twice.

With idempotency: only the first request is processed and all retries return the same response.

2) üí≥ Payment Capture / Payment Processing

Scenario: Payment API is retried due to gateway timeout or network issue.

Without idempotency: payment can be captured multiple times.

With idempotency: payment is executed only once and retries return the original payment result.

3) üì¶ Inventory Reservation / Stock Deduction

Scenario: Inventory reserve/deduct call is retried due to application crash or client retry.

Without idempotency: stock may be reduced multiple times causing incorrect inventory.

With idempotency: stock is deducted only once.

4) üîî Webhook Processing (Stripe / Razorpay / PayPal / Zapier)

Scenario: Webhook providers retry the same webhook event multiple times.

Without idempotency: duplicate invoice generation / duplicate order update / duplicate notifications.

With idempotency: use webhook eventId as key and process each event exactly once.

5) ‚öôÔ∏è Background Jobs / Queue Reprocessing (Kafka / SQS / RabbitMQ)

Scenario: Consumer crashes after processing but before ack, so the message is delivered again.

Without idempotency: the same job may run twice (double shipment / double state updates).

With idempotency: use messageId as key and safely reprocess.

6) üßæ Invoice Generation / Document Generation

Scenario: Invoice generation is retried due to failure or retry policy.

Without idempotency: multiple invoices can be created for the same order.

With idempotency: invoice is generated only once.

7) üì© Notification / Email / SMS Sending

Scenario: notification send is retried due to network issues.

Without idempotency: user receives duplicate emails/SMS.

With idempotency: only one notification is sent per unique key.

8) ü§ñ Zapier Integration (Automation Safety)

Scenario: Zapier workflows may trigger twice or retry because of network failures / step failures / replay.

Without idempotency: duplicate CRM updates, duplicate Slack messages, duplicate payments.

With idempotency: Zapier sends a stable idempotency key and your service blocks duplicate execution.

üí° Key Insight: This service identifies duplicate requests by tracking unique keys. When the same key arrives multiple times, it returns isDuplicate: true. Your application must use this signal to skip operations and return cached results, ensuring that the same operation is never executed twice.
‚ö†Ô∏è Best Practices:
  • Use stable IDs: Always use existing, immutable identifiers (Payment ID, Order ID, Event ID, Message ID) as keys
  • Check before execution: Call /check with your unique key BEFORE executing any operation
  • Cache on success: After successful first execution, cache the response locally
  • Return cached on duplicate: When isDuplicate: true is returned, immediately return your cached result without re-executing
  • Set appropriate TTL: Configure ttlSeconds based on your expected retry window (default: 3600 seconds / 1 hour)

üîç Check API

POST /idempotency/check üîê Auth Required

Atomically checks if an idempotency key exists and inserts it if not found or expired. This endpoint always returns 200 OK for successful checks (even for duplicates) to support integrations like Zapier that treat non-2xx status codes as errors.

Request Body

Field Type Required Description Constraints
idempotencyKey string Required Unique key to identify the operation Max 64 characters
clientId string Optional Client identifier for multi-tenant scenarios Max 64 characters
ttlSeconds long Optional Custom TTL in seconds (overrides default 3600s) Max 3600 seconds (1 hour)
‚ö†Ô∏è Validation Rules:
  • idempotencyKey: Must be between 1-64 characters. Use meaningful, unique identifiers (Payment ID, Order ID, etc.)
  • ttlSeconds: Maximum 3600 seconds (1 hour). Default is 3600s if not specified. This limits how long we remember processed operations.
  • If you need longer memory windows, contact support for enterprise plans.

Request Example

POST http://144.24.119.46:8080/idempotency/check
Content-Type: application/json
X-API-KEY: your-api-key-here

{
  "idempotencyKey": "payment-123-abc",
  "clientId": "client-A",
  "ttlSeconds": 3600
}

Response Body

Field Type Description
idempotencyKey string The idempotency key from the request
isDuplicate boolean false = new key (proceed), true = duplicate (skip)
createdAt timestamp When the key was first created (ISO 8601 format)
expiresAt timestamp When the key will expire (ISO 8601 format)
processingTimeNanos long Processing time in nanoseconds

Response Codes

200 OK All responses return 200 OK (check resultStatusCode in body for actual status)

‚ö†Ô∏è Important Behavior (Zapier Compatible):
  • HTTP Status: Always 200 OK for all responses (success, validation error, server error)
  • Actual Status: Check the resultStatusCode field in the response body:
    • 200 = Successful check (new or duplicate)
    • 400 = Invalid request body or validation error
    • 401 = Missing or invalid API key
    • 500 = Server error occurred
  • Check isDuplicate field to determine if operation should proceed
  • If isDuplicate is false, proceed with your operation
  • If isDuplicate is true, skip the operation (already processed)
  • Expired keys are automatically replaced with new entries

cURL Example

# Check idempotency (first request - new key)
curl -X POST http://144.24.119.46:8080/idempotency/check \
  -H "Content-Type: application/json" \
  -H "X-API-KEY: your-api-key" \
  -d '{
    "idempotencyKey": "payment-order-12345",
    "clientId": "user-abc",
    "ttlSeconds": 3600
  }'

# Response: {"isDuplicate": false, ...} - Proceed with operation

# Check same key again (duplicate)
curl -X POST http://144.24.119.46:8080/idempotency/check \
  -H "Content-Type: application/json" \
  -H "X-API-KEY: your-api-key" \
  -d '{
    "idempotencyKey": "payment-order-12345",
    "clientId": "user-abc"
  }'

# Response: {"isDuplicate": true, ...} - Skip operation

JavaScript/Fetch Example

async function checkIdempotency(key, clientId) {
  const response = await fetch('http://144.24.119.46:8080/idempotency/check', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'X-API-KEY': 'your-api-key'
    },
    body: JSON.stringify({
      idempotencyKey: key,
      clientId: clientId,
      ttlSeconds: 3600
    })
  });

  const result = await response.json();
  
  if (result.isDuplicate) {
    console.log('Duplicate request - skipping operation');
    return false;
  } else {
    console.log('New request - proceeding with operation');
    return true;
  }
}

import requests

def check_idempotency(key, client_id):

        'http://144.24.119.46:8080/idempotency/check',

            'X-API-KEY': 'your-api-key'

        print('Duplicate request - skipping operation')
        return False
    else:
        print('New request - proceeding with operation')
        return True

Integration Flow

Typical Integration Pattern:
  1. Generate a unique idempotency key for your operation (e.g., UUID, transaction ID)
  2. Call POST /idempotency/check with the key
  3. Check the isDuplicate field in the response:
    • If false: This is the first time seeing this key ‚Üí Proceed with your operation
    • If true: This key was already processed ‚Üí Skip the operation and return cached result
  4. The key will automatically expire after the TTL period
